# 图 3-4：脉冲式 on-off 攻击时序图

## 一、原理（为什么需要这张图）

### 论文中的理论基础

实际攻击者常采用脉冲式 on-off 策略：周期性开关攻击流量，以逃避基于持续高速率的检测。这张图验证 MS-SatShield 在这种动态攻击模式下的检测响应和吞吐保护能力。

论文第 236 行声称："rate-only 的 top-k 命中会出现间歇性掉出...引入持续性后 Recall 曲线更平滑。"

### 本图的核心职责

fig3_2(b) 展示了 FO/FI 在**静态退化**下的价值。本图承担展示 persistence 在**动态退化**下的价值：

- **静态退化**（fig3_2b）→ FO/FI 是解药（S0→S1 的提升）
- **动态退化**（fig3_4）→ persistence 是解药（S1→S2 的提升）

在静态场景中 persistence 无法提供区分能力（攻击 key 和良性重流 key 的 P̃ 均饱和到 ~1.0），但在动态 on-off 场景中，persistence 通过"记忆"功能维持对间歇性攻击 key 的检测。

### 攻击模型

- 攻击流量按固定周期（on_duration=10, off_duration=10 epoch）交替出现
- 每次 on 阶段发起 M=100 的退化攻击
- warmup 阶段（5 epoch）无攻击，用于建立基线

### S0/S1/S2 三方案对比

| 方案 | 权重 (α,β,γ) | 特征 |
|------|-------------|------|
| S0 (rate-only) | (1.0, 0.0, 0.0) | 仅用归一化速率 |
| S1 (+FO/FI) | (0.5, 0.5, 0.0) | 加入 fan-in/fan-out |
| S2 (+persist) | (0.5, 0.5, 0.2) | 加入持续性（γ 为加性奖励） |

> **注意**：S2 的权重采用“加性奖励”方案：α+β=1.0 构成基础检测分数，γ=0.2
> 为持续性额外加分（score 可 > 1.0）。这保证 S2 永远不会比 S1 差。

### 持续性扩展候选集机制

当 γ>0 时，`epoch_sim.py` 会将有持续性历史（P≥2）但不在当前 top-K 中的 key 也纳入评分候选集。这直接实现了论文的声明：

> "可疑 key 即便在少数窗口短暂跌出 top-k，其 P_t(a) 仍可维持较高水平。"

具体实现：
1. 每个 epoch 结束时，仅将 top-K 候选记录到 `PersistenceTracker`
2. 下一个 epoch 评分时，除 top-K 候选外，还检查历史候选中 P≥2 的 key
3. 这些"持续性扩展 key"使用聚合 peer 数（而非位图）估计 FO/FI
4. 扩展后的候选集一起参与评分和阈值判决

### 队列缓解模型

链路过载时，高优先级队列保留 `high_prio_bw_ratio × capacity`（默认 80%）给检测到的良性流，低优先级队列承受丢弃。

### 位置

第三章"结果分析"节，展示系统在动态攻击下的时域响应特性，特别是 persistence 特征的核心价值。

## 二、预期结论（图应该呈现什么趋势）

### (a) 攻击状态+链路利用率

- on 阶段利用率 >1.0（过载），off 阶段回到正常水平
- 攻击状态用 axvspan 红色带标注

### (b) top-k 命中数

- on 阶段迅速上升至接近攻击 key 总数（~100）
- off 阶段降为 0
- top-k 候选生成与评分方案无关（仅依赖速率排序），三种方案共用

### (c) Recall —— 三方案核心对比

三方案呈现截然不同的 Recall 行为：

1. **S0 (rate-only)**：M=100 下 rate-only 完全失效（per-key rate=100 Mb/s < rate_p99=484.9），Recall ≈ 0
2. **S1 (+FO/FI)**：FO/FI 特征使 score > τ，但 on-off 切换时 top-K 候选集抖动可能导致 Recall 波动
3. **S2 (+persist)**：持续性扩展候选集使得短暂跌出 top-K 的攻击 key 仍被检测到，Recall 曲线更平滑、更高

### (d) 良性吞吐 —— 缓解效果对比

- S0：检测失效，吞吐下降严重
- S1：检测生效但 on-off 切换处有波动
- S2：持续性平滑检测，吞吐恢复更稳定

### off-phase 行为

off 阶段无攻击 key，Recall 取空真值 1.0（无正类需检出）。三方案 off-phase 曲线重叠于 1.0。

## 三、当前实验数据

### on-phase 平均指标

| 方案 | on-phase 平均 Recall | on-phase 平均良性吞吐 |
|------|---------------------|---------------------|
| S0 (rate-only) | 0.010 | 0.572 |
| S1 (+FO/FI) | 0.990 | 0.881 |
| S2 (+persist) | **0.999** | **0.885** |

### S2 优势分析

S2 相比 S1 的改进来自两个机制：

1. **加性奖励权重**：S2 的 γ=0.2 是额外加分（而非从 α、β 中借权重），因此 S2 的基础检测能力与 S1 相同（α+β=1.0），持续性只会提升、不会降低检测性能。
2. **持续性扩展候选集**：当 γ>0 时，epoch_sim 会将历史上 P≥2 但未进入当前 top-K 的 key 也纳入评分，实现论文所述的“记忆”功能。

这使得 S2 在 on-off 场景下 Recall 从 S1 的 0.990 提升到 0.999，同时良性吞吐也略有提升（0.881→0.885）。

## 四、代码流程

### 入口

`experiments/ch3_ms_satshield/fig3_4_onoff.py` → `generate_fig3_4()`

### 核心处理步骤

```python
# 1. 加载数据，生成 benign_flows 和 attack_flows (M=100)

# 2. 构建 on-off schedule
#    warmup(5) + 周期性 on(10)/off(10)，共 100 epoch

# 3. 定义三种检测方案（使用 dataclasses.replace 创建不同权重配置）
#    S0: replace(MS_CFG, alpha=1.0, beta=0.0, gamma=0.0)
#    S1: replace(MS_CFG, alpha=0.5, beta=0.5, gamma=0.0)
#    S2: MS_CFG (α=0.5, β=0.5, γ=0.2, 加性奖励)

# 4. 对每种方案运行仿真（固定 random.seed(42) 保证公平对比）
#    run_multi_epoch_sim(benign, attack, capacity, schedule, cfg)
#    S2 方案在 epoch_sim 中自动启用持续性扩展候选集（因 γ>0）

# 5. 绘制 4 子图：
#    (a) 攻击状态 + 利用率（仅 S2，与方案无关）
#    (b) top-k 命中数（仅 S2，与方案无关）
#    (c) Recall —— S0/S1/S2 三条曲线
#    (d) 良性吞吐 —— S0/S1/S2 三条曲线
```

### 绘图参数

- 4 子图垂直排列，figsize=(10, 13)，共享 x 轴
- S0：灰色虚线，S1：蓝色点划线，S2：红色实线
- 攻击 on 区间用红色半透明带标注
- 输出：`experiments/figures/ch3/fig3_4_onoff.{pdf,png}`

## 五、修改历史

### 2026-02-26：三重修复 —— 持续性扩展候选集 + 加性奖励权重 + 职责重新定位

**问题**：
1. fig3_2(b) 的 S2 与 S1 重叠，persistence 在静态场景无区分能力
2. fig3_4 的 S2 反而低于 S1（0.863 vs 0.990），因为 (0.4,0.4,0.2) 权重分配导致冷启动
3. 论文声称 persistence 可维持跌出 top-K 的 key 的检测，但代码未实现

**修复**：
1. `epoch_sim.py` 新增持续性扩展候选集：当 γ>0 时，P≥2 的历史 key 纳入评分
2. `config.py` 权重从 (0.4,0.4,0.2) 改为 (0.5,0.5,0.2)：persistence 为加性奖励
3. fig3_2(b) 移除 S2，仅保留 S0 vs S1

**效果**：S2 on-phase 平均 Recall 从 0.863 提升到 0.999，高于 S1 的 0.990。

### 2026-02-26：添加 S0/S1/S2 三方案对比

**问题**：论文声称"引入持续性后 Recall 曲线更平滑"，但原图 3-4 只展示了一条曲线。

**修复**：使用 `dataclasses.replace` 创建三种权重配置，固定随机种子分别运行仿真。
