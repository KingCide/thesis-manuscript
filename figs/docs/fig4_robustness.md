# 图 4-robustness：碎片化攻击鲁棒性

## 一、原理（为什么需要这张图）

### 论文中的理论基础

碎片化攻击（退化攻击）是 LFA 的高级变种：攻击者将流量分散到更多 decoy 目的地（M 增大），使每个 key 的聚合速率降低，混入良性流量长尾。这张图评估各方案在碎片化攻击下的鲁棒性。

### 碎片化的影响

- 每个 decoy 的聚合速率降低 → rate-only 检测失效
- 攻击的 FO/FI 和持续性特征受稀释但仍可区分 → 多签名方案有效
- CoopSatShield 通过推回缓解降低无效带宽 → 吞吐恢复更优

### 三方案定位

| 方案 | 检测方式 | 缓解方式 | 核心优势 |
|------|---------|---------|---------|
| SatShield | rate-only（S0 权重） | 本地队列调度 | 简单快速 |
| MS-SatShield | 多签名 rate+FO/FI+persist | 本地队列调度 | 对退化攻击鲁棒 |
| CoopSatShield | 多签名（同 MS-SatShield） | gossip + 推回 + 队列 | 吞吐恢复更优 |

### 位置

第四章"对碎片化攻击证据的鲁棒性"节。

## 二、F1 倒 U 型趋势解释

### 预期趋势

F1 随 M 呈现**倒 U 型**（非单调递减），峰值在 M≈50-100。**这与第三章完全一致**。

### 三阶段行为

#### 1. 精度受限段（M≤10）：Recall=1.0，但 F1 低

M 小时攻击 key 数少（如 M=1 仅 1 个），但检测器恒定产生 ~19 个良性重流 FP（rate>242 Mb/s，即 > tau × rate_p99）。**攻击 key 全部被正确检出（Recall=1.0）**，F1 低仅因 Precision 被 FP 拖累：

- M=1: tp=1, fp=19 → P=1/20=0.05, R=1.0, **F1=0.095**
- M=5: tp=5, fp=19 → P=5/24=0.21, R=1.0, **F1=0.345**
- M=10: tp=10, fp=19 → P=10/29=0.34, R=1.0, **F1=0.513**

**关键**：F1 低≠检测失败。M=1 时唯一的攻击 key（rate=10,024 Mb/s）被正确检出，吞吐恢复至 0.886（SS/MS）或 1.000（CS）。

#### 2. 有效检测段（M=50~100）

攻击 key 数 >> FP 数，FO/FI 特征有效区分攻击与良性：

- M=50: MS-SatShield P=0.714, R=1.0, F1=0.833
- M=100: MS-SatShield P=0.833, R=1.0, **F1=0.909**（峰值）

此段 SatShield 失效：M=50 时攻击 key 单 key 速率 ~200 Mb/s < tau×rate_p99=242.5 → rate-only 无法检出。

#### 3. 召回受限段（M≥500）

攻击 key 单 key 速率 ≤20 Mb/s，落入良性分布区间。大量攻击 key 脱离 top-K 候选集（TopK_Recall=0.29~0.30）：

- M=500: MS-SatShield P=0.936, R=0.292, F1=0.445
- M=1000: MS-SatShield P=1.000, R=0.304, F1=0.466

注意：Precision 在此段反而很高（被选入 top-K 的攻击 key 几乎全部被正确识别），但 Recall 低导致 F1 下降。

### ~19 个恒定 FP 的来源

来自真实良性重流 key（如热门目的地），rate=474~606 Mb/s，均超过 tau×rate_p99=242.5 Mb/s。这是所有基于阈值检测器的固有下限，与第三章 fig3_2(b) 分析一致。在实际部署中，队列缓解是渐进式的，~19 个被轻微降级的良性 key 是可接受的代价。

## 四、CoopSatShield F1 = MS-SatShield 的原因

### 现象

CoopSatShield 与 MS-SatShield 的 F1 在所有 M 值下完全相同。

### 根因

当前仿真架构采用**全局检测**（性能优化）：所有流量在全局统一聚合、top-K 筛选、FO/FI 估计、评分。CoopSatShield 和 MS-SatShield 使用完全相同的检测流水线（`MultiSignatureScorer`），因此检测结果相同。

gossip 协议在仿真中分发的是全局检测结果，而非真正的分布式节点证据，因此无法提供额外的检测增益。

### CoopSatShield 的真实优势

CoopSatShield 的优势完全体现在**缓解层面**：

1. **推回降低 W_waste**：将丢弃点从受害链路前移到攻击源附近（R_up=3 跳），减少中间链路的无效带宽占用
2. **吞吐恢复更优**：M≤100 时 CoopSatShield 吞吐=1.000（vs MS-SatShield=0.886），因为推回主动释放了链路带宽

### 论文叙事建议

- **检测能力**：CoopSatShield 的检测等效于 MS-SatShield（同一检测算法）
- **缓解能力**：CoopSatShield 通过 gossip 触发推回，将攻击流在上游丢弃，释放链路带宽
- **核心贡献**：协同不在于"检测更好"，而在于"缓解更好"——吞吐恢复 + W_waste 降低

## 五、代码流程

### 入口

`experiments/ch4_coopsatshield/fig4_robustness.py` → `generate_fig4_robustness()`

### 核心处理步骤

```python
# fig4_robustness.py 主流程:

# 对每个 M in M_range=[1,5,10,50,100,500,1000]:
#   a. generate_degraded_attack_flows(M=M) → 退化攻击流
#   b. 对 3 种方案 (SatShield / MS-SatShield / CoopSatShield):
#      - 运行检测 + 缓解
#      - 记录 F1, Recall, Precision, throughput, W_waste
#   c. 汇总结果

# 2×2 子图:
# (a) F1（实线）+ Recall（虚线）vs M
# (b) Precision vs M
# (c) 良性吞吐 vs M
# (d) W_waste vs M
```

### 关键参数

| 参数 | 值 | 说明 |
|------|---|------|
| 聚合 n_noise_keys | 0 | 不注入噪声，与第三章对齐 |

## 六、修改历史

### 2026-02-27：添加 Recall/Precision 子图，改为 2×2 布局

**问题**：
原 1×3 布局仅展示 F1，M=1 时 F1=0.095 被误读为"检测失败"，但实际 Recall=1.0（攻击 key 全部检出）。F1 低仅因 ~19 个良性重流 FP 压低 Precision。

**修复**：
1. 布局从 1×3 改为 2×2，新增 (a) Recall 虚线叠加和 (b) Precision 子图
2. F1 子图标注"Recall=1.0, ~19 FP 压低 F1"，避免误读
3. 输出同时打印 F1、Recall、Precision，方便验证
4. 良性吞吐 y 轴从 0 改为 0.5，放大差异

### 2026-02-26：修复噪声 key 污染 + 固定 K（与第三章 P1-1 同源 bug）

**问题**：
1. fig4_robustness 的 F1 在 M=1 时所有方案 <0.1，且 F1 随 M 增大反而上升，看似与第三章"M 越大检测越难"矛盾
2. CoopSatShield F1 < MS-SatShield（如 M=50: 0.543 vs 0.741）

**根因**：ch4 的 `baselines.py` 和 `coop_engine.py` 存在与第三章 P1-1 相同的两个 bug：

1. **噪声 key 污染**：`aggregate_by_key(flows)` 使用默认 `n_noise_keys=5000`，注入 5000 个背景噪声 key。其中 77 个噪声 key 的 rate > tau×rate_p99 = 242 Mb/s，恒定被误检为 FP，严重压低 precision。
2. **固定 K=1000**：使用 `cfg.K=1000`（固定值），在 5377 个 key 中取 top-1000 → 96% 是噪声 key。高 M 时攻击 key 被噪声挤出 top-K（TopK_Recall: 0.080 vs 修复后 0.292）。

**修复**：

1. `baselines.py`：`aggregate_by_key(flows)` → `aggregate_by_key(flows, n_noise_keys=0)`
2. `baselines.py`：添加 `_adaptive_K()` 函数，`K = clip(0.3 * N_keys, 50, 1000)`
3. `coop_engine.py`：同样修复聚合和 K 值
4. `coop_engine.py`：F1 计算改用 `effective_detected`（本地+推回），体现协同检测

**修复后**：F1 趋势与第三章完全一致（SatShield ↔ S0，MS-SatShield ↔ S1 数值匹配）。

**关于"F1 倒 U 型与第三章矛盾"**：实际并无矛盾。第三章的 S1 F1 也是倒 U 型（M=1: 0.091 → M=100: 0.908 → M=1000: 0.420）。低 M 处 F1 低是因为 ~19 个恒定 FP 压低了 Precision（TP 太少），不是因为检测失效。"M 越大检测越难"指的是 M>100 后 Recall 下降的阶段。
